# Story 1.4 — Registro de Presença com Validação (IP + Geo + Horário)

**Status:** Ready for Review
**Épico:** 1 — Parte 1
**Depende de:** 1.1, 1.3

## User Story
Como **aluno autenticado dentro da universidade**,
Quero **escanear o QR Code da aula e registrar minha presença**,
Para que **minha frequência seja contabilizada apenas quando eu estou realmente no local e horário corretos**.

## Acceptance Criteria

- [x] AC1: Endpoint `GET /presenca?id=<aula_id>&token=<token>` renderiza a página de registro.
- [x] AC2: Se o aluno não estiver logado, allauth redireciona para login e volta à URL original após auth.
- [x] AC3: Página solicita permissão de geolocalização via `navigator.geolocation.getCurrentPosition()`.
- [x] AC4: Endpoint `POST /api/presenca` recebe `{ aula_id, token, latitude, longitude }` e executa, em ordem, as validações:
  1. **Autenticação:** usuário logado e com role `ALUNO` (caso contrário 403).
  2. **Token válido:** `Aula.token_qrcode == token` recebido (caso contrário 400 — "QR inválido").
  3. **Horário:** `data`+`horario_inicio` ≤ `now` ≤ `data`+`horario_fim` (caso contrário 400 — "Aula não está em andamento").
  4. **Duplicidade:** não existe `Presenca(aluno=user.aluno, aula=aula)` (caso contrário 409 — "Presença já registrada").
  5. **Rede:** IP de origem (`request.META['HTTP_X_FORWARDED_FOR']` ou `REMOTE_ADDR`) dentro de `UNIVERSITY_IP_RANGES` (lista de CIDRs em `settings`) (caso contrário 403 — "Fora da rede institucional").
  6. **Geolocalização:** distância haversine entre (lat, lng) do aluno e (lat, lng) da sala ≤ `sala.raio_permitido` (caso contrário 403 — "Fora do raio permitido").
- [x] AC5: Aprovado em tudo → cria `Presenca` com `status='PRESENTE'`, `ip_registrado`, coordenadas, `horario_registro=now()`.
- [x] AC6: `UNIVERSITY_IP_RANGES` configurável em `.env` (default dev: `127.0.0.0/8,192.168.0.0/16,10.0.0.0/8`).
- [x] AC7: Função `is_ip_in_ranges(ip, ranges)` usando `ipaddress` stdlib, com testes unitários.
- [x] AC8: Função `haversine_distance(lat1, lng1, lat2, lng2)` retornando metros, com teste unitário.
- [x] AC9: Tela de sucesso / erro com mensagem clara para o aluno.
- [x] AC10: Mensagens de erro não vazam detalhes internos (apenas motivo amigável).

## Notas Técnicas
- Views em FBV simples (`def presenca_view(request)`), sem DRF ou CBVs complexas nesta etapa (DRF é Parte 2).
- Usar apenas `ipaddress` e `math` da stdlib — nada de libs externas de geo.
- JS: Fetch API + `{% csrf_token %}` no template (sem frameworks).
- `LogEntry.objects.log_action()` ao registrar presença, para auditoria.

### Issues

**🟡 Issue-1 (MEDIUM) — Role check frouxo**
- **Local:** `_get_aluno_for_user` usa `hasattr(user, 'aluno')` em vez de validar Group `Alunos`.
- **Impacto:** usuário com `Aluno` linkado mas **fora** do Group `Alunos` consegue registrar presença. PDF diz "autenticação do aluno" + Story AC4.1 diz "role `ALUNO`".
- **Fix:** `aluno is not None and _is_aluno(user)` — 1 linha.

**🟡 Issue-2 (MEDIUM) — Race condition no check de duplicidade**
- **Local:** `_validate_presenca` L434 → `.exists()` seguido de `.create()` (não atômico).
- **Mitigação existente:** `unique_together(aluno, aula)` no modelo garante integridade no DB. Porém 2º request concorrente gera `IntegrityError` → HTTP 500 em vez de 409.
- **Fix:** envolver `create()` em `try/except IntegrityError` retornando 409.

**🟢 Issue-3 (LOW) — Comparação de token não constant-time**
- **Local:** `str(aula.token_qrcode) != str(token)` (L424)
- **Nota:** UUID não é segredo guessable; risco teórico. Hábito seguro seria `secrets.compare_digest(...)`.

**🟢 Issue-4 (LOW) — Import local `from datetime import datetime as _dt`**
- **Local:** dentro de `_validate_presenca`. Funciona, mas pode subir para o topo do módulo.
