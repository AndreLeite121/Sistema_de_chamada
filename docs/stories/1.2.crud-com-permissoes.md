# Story 1.2 — CRUD das Entidades com Permissões por Role

**Status:** Ready for Review
**Épico:** 1 — Parte 1
**Depende de:** 1.1

## User Story
Como **usuário autenticado**,
Quero **poder criar, listar, visualizar, editar e excluir as entidades do sistema conforme meu papel**,
Para que **cada perfil (Admin/Professor/Aluno) tenha acesso apropriado às operações permitidas pelo roteiro**.

## Acceptance Criteria

- [x] AC1: Generic CBVs do Django (`ListView`, `DetailView`, `CreateView`, `UpdateView`, `DeleteView`) para as 6 entidades. Sem mixins customizados, sem abstrações extras.
- [x] AC2: Templates usando `base.html` existente. `ModelForm` padrão do Django (sem django-crispy-forms ou similar).
- [x] AC3: Filtros simples via `request.GET` no `get_queryset()` da `ListView` (sem django-filter). Campos: aluno, disciplina, professor, data.
- [x] AC4: Permissões via:
  - `LoginRequiredMixin` em todas as views.
  - `PermissionRequiredMixin` usando as permissions nativas (`core.add_aula`, `core.change_presenca` etc.) geradas automaticamente pelos modelos.
  - `get_queryset()` sobrescrito para filtrar por dono (professor vê só suas aulas; aluno vê só suas presenças).
- [x] AC5: Exclusão restrita a admin (staff) ou ao dono — verificado com `UserPassesTestMixin`.
- [x] AC6: Edição de `Presenca` apenas pelo professor da aula (via `UserPassesTestMixin`).
- [x] AC7: Toda view que escreve/deleta/cria chama `LogEntry.objects.log_action(...)` para registrar auditoria (Story 1.1 AC7).
- [x] AC8: Menu `navbar.html` usa `{% if perms.core.add_aula %}` etc. — condicional por permission nativa.
- [x] AC9: Tentativas não autorizadas retornam HTTP 403 (comportamento padrão dos mixins).

## Notas Técnicas
- **Zero abstrações novas.** Nada de `RoleRequiredMixin`, `OwnerOrAdminMixin`. Usar mixins/permissions do próprio Django.
- Admin do Django já cobre CRUD completo para admins — as views CRUD do `core` existem principalmente para Professor e Aluno.

### Issues

**🔴 Issue-1 (HIGH) — AC9 falhando: permissão negada redireciona em vez de retornar 403**
- **Local:** `core/views.py` — todas as views com `PermissionRequiredMixin`
- **Root cause:** Default do mixin é `raise_exception=False` → redireciona ao login
- **Fix sugerido:** adicionar `raise_exception = True` em uma base class ou setar `handler403` no `urls.py`
- **Impacto:** usuário autenticado sem permissão vai para /accounts/login/ (UX confusa); testes que esperam 403 falham

**🔴 Issue-2 (CRITICAL) — Ownership gap em PresencaDelete**
- **Local:** `core/views.py:321-324`
- **Risco:** Professor A com permissão `core.delete_presenca` pode deletar presença de aula de Professor B (mesmo Groups padrão não dão essa perm, mas o design é frágil — admin que adicionar perm manualmente abre brecha)
- **Fix:** adicionar `UserPassesTestMixin` + `test_func` igual ao `PresencaUpdate`
- **Recomendação análoga:** `DisciplinaDelete` também deveria ter ownership check para simetria

**🟡 Issue-3 (MEDIUM) — Queries com `|` sobre QuerySets**
- **Local:** `AlunoList.get_queryset` L61; `DisciplinaList.get_queryset` L147
- **Problema:** `qs.filter(A) | qs.filter(B)` faz UNION de dois querysets já avaliados separadamente; idiomático e mais seguro é `from django.db.models import Q` e `qs.filter(Q(A) | Q(B))`
- **Fix:** refatorar para `Q()`

**🟡 Issue-4 (MEDIUM) — Paginação perde filtros**
- **Local:** `_pagination.html`
- **Problema:** `?page=N` descarta `q`, `data`, `disciplina` na navegação
- **Fix:** propagar querystring — `?page=N&{{ request.GET.urlencode }}` com sanitização

**🟡 Issue-5 (MEDIUM) — Sem testes automatizados**
- **Local:** `core/tests.py` é stub
- **Impacto:** AC não verificáveis em CI; regressões não detectáveis
- **Sugestão mínima:** `test_views.py` com smoke tests — aluno/professor/admin fazendo login e acessando list/create; validar 403 em não-autorizados

**🟢 Issue-6 (LOW) — Filter por GET sem sanitização**
- **Local:** `filter(professor_id=prof)` aceita string não-numérica → ValueError 500
- **Fix:** `if prof and prof.isdigit(): qs = qs.filter(...)`
