# Story 1.5 — Dashboard Personalizado + Histórico de Alterações

**Status:** Ready for Review
**Épico:** 1 — Parte 1
**Depende de:** 1.1, 1.2, 1.4

## User Story
Como **usuário autenticado**,
Quero **uma página principal e um dashboard com visão adequada ao meu papel**,
Para que **eu acesse rapidamente as informações e ações relevantes para mim**.

## Acceptance Criteria

- [x] AC1: Rota `/` redireciona autenticados para `/dashboard/` e anônimos para `/accounts/login/` (usar `LOGIN_URL` já padrão do Django).
- [x] AC2: **Uma única `dashboard_view`** (FBV) e **um único template `dashboard.html`** com blocos condicionais `{% if user.groups... %}` para Admin/Professor/Aluno. Sem três templates separados.
- [x] AC3: Conteúdo por role (mesmo template, blocos diferentes):
  - Admin: contadores gerais (User, Disciplina, Aula hoje, Presenca hoje) via `Model.objects.count()`.
  - Professor: suas disciplinas, próximas aulas, últimas presenças de suas aulas.
  - Aluno: disciplinas matriculadas, próxima aula, total de presenças, % de frequência.
- [x] AC4: Página `/auditoria/` (restrita a staff via `@staff_member_required`) lista últimos `LogEntry` do Django com filtro por usuário, tipo de ação e período. Paginação com `Paginator`.
- [x] AC5: Sem gráficos (Parte 2). Apenas tabelas e cards de contagem.
- [x] AC6: Navbar condicional usando `{% if user.is_authenticated %}` + `{% if perms %}`.
- [x] AC7: Queries com `select_related` onde for óbvio (FK em ListView).

## Notas Técnicas
- Frequência do aluno: `count(presencas) / count(aulas_da_disciplina_ja_ocorridas)`.
- Auditoria: paginação simples (`Paginator`, 50 por página).

### Issues

**🟡 Issue-1 (MEDIUM) — N+1 no cálculo de frequência do aluno**
- **Local:** `dashboard_view` L56-61 — loop por disciplina fazendo 2 counts cada.
- **Impacto:** aluno em 10 disciplinas = 21 queries. Tolerável, mas ineficiente.
- **Fix sugerido:** usar `annotate(total=Count('aulas', filter=Q(aulas__data__lte=hoje)), presentes=Count('aulas__presencas', filter=Q(aulas__presencas__aluno=aluno, aulas__presencas__status='PRESENTE')))` em 1 query.

**🟢 Issue-2 (LOW) — Filtro "modelo" na auditoria case-insensitive, mas exige nome exato**
- **Local:** `auditoria_view` L14: `filter(content_type__model=modelo.lower())`
- **Nota:** `content_type.model` é sempre lowercase, então `.lower()` é ok. Mas o UX do form pede nome do modelo ("ex.: aula") — usuários leigos podem errar. Sem fix necessário, apenas help-text.

**🟢 Issue-3 (INFO) — Tabela de auditoria não mostra IP do ator**
- `LogEntry` nativo não captura IP. Se quiser IP na auditoria, precisaria de um modelo próprio (fora do escopo — PDF não exige IP na auditoria, apenas em Presença).
